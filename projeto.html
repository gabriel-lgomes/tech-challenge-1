O que é cada ferramenta? (explicado para devs frontend)

🐍 Python

A linguagem usada para Machine Learning. No mundo frontend você usa JavaScript; no mundo de dados, Python é o equivalente.

Analogia: Python é o JavaScript do mundo de dados — mesma ideia, ecossistema diferente.

🐼 Pandas

Biblioteca para manipular dados em tabelas (linhas e colunas). Lê o CSV do dataset e permite filtrar, agrupar e transformar os dados.

Analogia: É como um Excel programável. Um DataFrame do Pandas é uma tabela — como uma array de objetos no JS, mas muito mais poderosa.

📊 Matplotlib / Seaborn

Bibliotecas para criar gráficos: histogramas, boxplots, heatmaps, etc. Tudo que foi gerado em reports/figures/ veio delas.

Analogia: São o D3.js / Chart.js do Python — só que você escreve menos código para gráficos científicos.

⚙️ Scikit-learn

A principal biblioteca de Machine Learning. Tem prontos todos os algoritmos que vamos usar: Regressão Logística, Random Forest, KNN, além de ferramentas de avaliação.

Analogia: É como um framework (tipo React) — você não cria os algoritmos do zero, só usa o que já está pronto e bem testado.

🔍 SHAP

Biblioteca de explicabilidade. Depois de treinar o modelo, ela mostra por que o modelo tomou cada decisão — quais variáveis mais pesaram no diagnóstico.

Analogia: É como o DevTools do modelo — você consegue ver "por dentro" o que influenciou cada previsão.

📦 Conda / Anaconda

Gerenciador de ambiente e pacotes para Python. Cria ambientes isolados para cada projeto, para que as dependências de um não quebrem o outro.

Analogia: É o npm + nvm do Python. O conda create -n projeto é igual ao nvm use 18 — isola a versão e os pacotes por projeto.

📓 Jupyter Notebook

Ambiente interativo onde você mistura código Python com texto explicativo e gráficos na mesma página. É o formato padrão em ciência de dados.

Analogia: É como um Storybook — você documenta e demonstra ao mesmo tempo. Cada "cell" é um bloco de código que você roda individualmente.

🏥 Dataset — Breast Cancer Wisconsin

Base de dados pública com 569 pacientes. Cada linha é uma biópsia com 30 medidas do tumor. A coluna alvo diz se era maligno ou benigno.

Analogia: É como um JSON com 569 objetos, cada um com 30 propriedades numéricas e um campo diagnosis: "M" | "B".

O que o projeto realmente faz?

A ideia central

O modelo aprende com histórico de biópsias anteriores (onde já se sabe o diagnóstico real) e, a partir disso, consegue prever o diagnóstico de uma nova biópsia que ele nunca viu.

Entrada: 30 medidas do tumor (tamanho, textura, formato, etc.)
Saída: Maligno (M) ou Benigno (B)

Analogia: É como treinar um sistema de autocomplete. Você mostra milhares de exemplos corretos, ele aprende o padrão e começa a acertar sozinho em casos novos.

Por que Recall é a métrica mais importante?

Em contexto médico, existem dois tipos de erro:

Falso Negativo: dizer que o tumor é benigno quando é maligno. Gravíssimo — a paciente vai para casa sem tratamento.
Falso Positivo: dizer que é maligno quando é benigno. Ruim — causa ansiedade e exames desnecessários, mas dá para resolver.

Recall mede justamente nossa capacidade de não perder nenhum caso maligno. Queremos que seja o mais alto possível.

Analogia: É como validação de formulário — você prefere mostrar um erro desnecessário (falso positivo) a deixar um dado inválido passar (falso negativo).

O que a Análise Exploratória (EDA) revelou?

✓ Dataset limpo

Nenhum valor ausente, nenhuma duplicata. O dataset do Kaggle já vem bem preparado — menos trabalho de limpeza.

Distribuição de diagnósticos

357 benigno (63%) e 212 maligno (37%). Há um leve desbalanceamento — mais casos benignos do que malignos. Isso será tratado na modelagem.

Features mais importantes para o diagnóstico

As variáveis com maior correlação com tumores malignos são: concave points, perimeter, radius e area — ou seja, tumores maiores e mais irregulares tendem a ser malignos. Isso é consistente com o que médicos sabem clinicamente.

⚠️ Alta correlação entre features

Área, raio e perímetro são quase a mesma informação (matematicamente correlacionadas). Isso se chama multicolinearidade e precisa ser tratada no pré-processamento — caso contrário, pode afetar a qualidade do modelo.

Analogia: É como ter três propriedades no mesmo objeto que sempre têm o mesmo valor. Você não precisa das três — uma resolve.

O que acontece nas próximas etapas?

Etapa 3 — Pré-processamento

Preparar os dados para os modelos:

• Remover a coluna id (não tem valor preditivo)
• Converter diagnosis: B → 0, M → 1 (modelos trabalham com números)
• Normalizar as features (colocar tudo na mesma escala)
• Separar os dados em treino (80%) e teste (20%)

Analogia: É como normalizar e validar os dados antes de fazer um POST para uma API — o modelo só aceita entrada em formato correto.

Etapa 4 — Modelagem

Treinar dois modelos e comparar:

• Regressão Logística: modelo simples, rápido e muito interpretável. Bom como baseline.
• Random Forest: conjunto de várias "árvores de decisão" votando juntas. Mais poderoso e gera feature importance automaticamente.
• Depois: SHAP para explicar cada previsão individualmente.
